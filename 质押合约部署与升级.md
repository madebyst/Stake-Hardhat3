# MetaNodeStake 部署与升级详解

## 一、部署流程

### 1.1 部署架构概览

部署分为三个阶段：奖励代币 → 质押合约（UUPS 代理） → 资金池配置。

```
┌─ 阶段 1 ─────────────────────────────────────────────┐
│ MetaNodeToken 部署                                     │
│   · 部署 ERC20 合约                                    │
│   · deployer 获得全部 1000 万 MetaNode                  │
│   · Etherscan 验证                                     │
└──────────────────┬────────────────────────────────────┘
                   │
                   ▼
┌─ 阶段 2 ─────────────────────────────────────────────┐
│ MetaNodeStake UUPS 代理部署                            │
│   · upgrades.deployProxy() 部署代理 + 实现合约          │
│   · initialize(MetaNode地址, startBlock, endBlock,      │
│                MetaNodePerBlock)                       │
│   · deployer 获得三个角色                               │
│   · 全部 MetaNode 从 deployer 转入质押合约              │
│   · 验证实现合约                                       │
└──────────────────┬────────────────────────────────────┘
                   │
                   ▼
┌─ 阶段 3 ─────────────────────────────────────────────┐
│ 添加质押池                                             │
│   · addPool(address(0), weight, ...) → ETH 池 (pid=0) │
│   · addPool(erc20Addr, weight, ...) → ERC20 池 (pid=1)│
│   · 部署 TestERC20 测试币（可选）                        │
└───────────────────────────────────────────────────────┘
```

### 1.2 阶段 1：部署奖励代币

```javascript
// scripts/deploy.js
const MetaNodeToken = await ethers.getContractFactory('MetaNodeToken');
const metaNodeToken = await MetaNodeToken.deploy();
await metaNodeToken.waitForDeployment();
```

MetaNodeToken 是一个最简 ERC20，构造函数中 `_mint(msg.sender, 10000000 * 1e18)`，一瞬间将所有代币铸给部署者。没有额外逻辑，也没有 mint 接口——总量永不变。

### 1.3 阶段 2：部署 UUPS 代理

这是最关键的一步，使用 OpenZeppelin Hardhat Upgrades 插件完成。

#### 1.3.1 deployProxy 的三个入参

```javascript
const stake = await upgrades.deployProxy(
    MetaNodeStake,                                    // ① 实现合约工厂
    [                                                  // ② initialize() 的参数数组
        metaNodeTokenAddress,    // _MetaNode
        startBlock,              // _startBlock
        endBlock,                // _endBlock
        metaNodePerBlock         // _MetaNodePerBlock
    ],
    {                                                  // ③ 选项
        initializer: "initialize",                    // 初始化函数名
        kind: "uups"                                  // 代理类型
    }
);
```

**① 工厂（不是合约实例）**：

`MetaNodeStake` 来自 `getContractFactory()`，手里有 ABI + Bytecode。Bytecode 是编译产物，存在本地 `artifacts/` 目录中。`deployProxy` 要用 Bytecode 部署新的实现合约到链上，只有工厂才有这个东西。`getContractAt()` 返回的是已部署合约的"遥控器"（只有地址 + ABI，没有 Bytecode），用它来部署是不可能的。

```
getContractFactory("MetaNodeStake")
  └→ 读取 artifacts/contracts/MetaNodeStake.sol/MetaNodeStake.json
     ├── ABI        ✅  函数接口
     └── Bytecode   ✅  编译产物，还没上链

getContractAt("MetaNodeStake", "0x...")
  ├── ABI     ✅
  ├── 地址    ✅  0x56682...
  └── Bytecode ❌  不需要了
```

**② 参数数组**：必须严格按照 `initialize()` 的形参顺序传入。

```solidity
function initialize(
    IERC20 _MetaNode,        // ← 数组[0]
    uint256 _startBlock,     // ← 数组[1]
    uint256 _endBlock,       // ← 数组[2]
    uint256 _MetaNodePerBlock // ← 数组[3]
) public initializer { ... }
```

**③ 选项**：

| 字段 | 值 | 含义 |
|------|-----|------|
| `initializer` | `"initialize"` | 部署后立刻调用的初始化函数名 |
| `kind` | `"uups"` | 使用 UUPS 代理（升级函数在实现合约中） |

注意 `MetaNodeStake.js` 中**没写 `kind: "uups"`**，插件默认会用 Transparent Proxy，与 `deploy.js` 行为不同。`deploy.js` 的写法是正确的标准版本。

#### 1.3.2 代理与实现如何"绑定"

`deployProxy` 一行干了三件事：部署实现 → 部署代理 → 绑定并初始化。

**步骤 1：部署实现合约**

插件用工厂里的 Bytecode 发交易到链上，得到 `implAddress`。此时实现合约孤零零地存在，没有任何代理指向它。

**步骤 2：部署 ERC1967 代理合约**

插件部署一个标准 ERC1967 代理，得到 `proxyAddress`。

**步骤 3：绑定 —— 在代理的存储里写入实现地址**

"绑定"本质上就是在代理合约的一个**固定存储槽位**里写入实现地址。这个槽位由 ERC1967 标准规定，所有人都知道它在哪：

```solidity
// ERC1967 标准：实现合约地址存在这个固定的 slot 里
bytes32 internal constant IMPLEMENTATION_SLOT =
    0x360894a13ba1a3210667c828492db98dca3e2076cc3735a920a3ca505d382bbc;
```

```
Proxy 存储布局:
┌──────────────────────────────────────────┐
│ Slot: ERC1967      │ 0x27ad... ← 实现地址  │ ← 一个 SSTORE 搞定绑定
│ Slot: 0            │ startBlock           │ ← initialize() 写入
│ Slot: 1            │ endBlock             │
│ Slot: 2            │ MetaNodePerBlock     │
│ ...                │ ...                  │
└──────────────────────────────────────────┘
```

不是合约间互相引用，不是事件，不是 mapping —— 就是普普通通一次 `SSTORE`。

#### 1.3.3 初始化走构造函数，后续调用走 fallback

代理和实现绑定后，插件立刻触发 `initialize()`，但**初始化调用和后续用户调用走的是两条不同的路径**：

**初始化（仅一次）—— 在 Proxy 构造函数里直接 `delegatecall`**：

```solidity
// ERC1967Proxy 的构造函数（简化版）
constructor(address _implementation, bytes memory _data) {
    // 写入实现地址到标准槽位
    ERC1967Utils.setImplementation(_implementation);
    // 直接 delegatecall 执行 initialize()
    if (_data.length > 0) {
        (bool success, ) = _implementation.delegatecall(_data);
        require(success, "initialization failed");
    }
}
```

`_data` 是 `initialize(params...)` 的 ABI 编码，插件在 JS 端就编码好了传进去。因为是 `delegatecall`，initialize 里修改的状态全部写在了 Proxy 自己身上：

```
initialize() 在代理上产生的效果:
  → startBlock       = 6529999
  → endBlock         = 9529999
  → MetaNodePerBlock = 0.02e18
  → MetaNode         = 0x5FbDB...
  → _grantRole(ADMIN_ROLE    → deployer)
  → _grantRole(UPGRADE_ROLE  → deployer)
  → _grantRole(DEFAULT_ADMIN → deployer)
```

**后续调用（永久）—— fallback 接管**：

初始化之后，用户调用 `deposit()`、`unstake()`、`claim()` 等，走的是另一个入口：

```solidity
// Proxy 的 fallback（简化版）
fallback() external payable {
    address _impl = ERC1967Utils.getImplementation();  // ① 从固定槽位读实现地址
    assembly {
        calldatacopy(0, 0, calldatasize())             // ② 原封不动拷贝 calldata
        let result := delegatecall(gas(), _impl, 0, calldatasize(), 0, 0)  // ③ 转发
        returndatacopy(0, 0, returndatasize())         // ④ 原封不动返回结果
        ...
    }
}
```

```
                    Proxy 的生命周期
                    ═══════════════

   构造函数（一次性）                     fallback（永久）
   ┌──────────────────┐               ┌──────────────────┐
   │ Proxy 刚诞生       │               │ Proxy 已经在链上   │
   │                   │               │                  │
   │ constructor 里    │               │ 收到不认识的方法   │
   │ 显式 delegatecall │               │ → fallback 捕获   │
   │ initialize()      │               │ → 读槽位拿实现地址 │
   │                   │               │ → delegatecall    │
   └──────────────────┘               └──────────────────┘
```

所以是**两扇门**：构造函数负责一次性初始化，fallback 接管此后所有用户交互。

#### 1.3.4 完整调用链路

```
用户调用 → stake.depositETH()
              │
              ▼
         Proxy (0x56682...) 收到调用
              │ 匹配不到 depositETH 这个函数
              │
              ▼
         fallback() 捕获
              │
              │  SLOAD(IMPLEMENTATION_SLOT)
              │  → 拿到 0x27ad...
              │
              │  delegatecall → Impl.depositETH()
              │                  ↑ 代码在 Impl 里执行
              │                    但读写全是 Proxy 的存储
              │
              ▼
       返回结果给用户
```

#### 1.3.5 关键部署参数

| 参数 | 实际值（Sepolia） | 含义 |
|------|-------------------|------|
| `_MetaNode` | `0x5FbDB...` | MetaNode 代币地址 |
| `_startBlock` | `6529999` | 质押开始区块 |
| `_endBlock` | `9529999` | 质押结束区块（约 30 天后） |
| `_MetaNodePerBlock` | `20000000000000000` | 每区块奖励 0.02 MetaNode |

**为什么用 `initialize()` 而不是 `constructor()`**：`constructor` 只在实现合约部署时执行一次，其状态修改不会影响代理。只有通过代理调用 `initialize()`，状态才写在代理的存储里，才能被后续所有 `delegatecall` 看到。

### 1.4 资金注入

```javascript
// 部署者将全部 MetaNode 转入质押合约作为奖励池
const tokenAmount = await metaNodeToken.balanceOf(signer.address);
await metaNodeToken.connect(signer).transfer(stakeAddress, tokenAmount);
```

部署脚本把 1000 万 MetaNode 全部转入了质押合约。系统没有 mint 能力，所以这 1000 万就是奖励池的总上限。耗尽之后，`_safeMetaNodeTransfer` 余额不足时将只转出剩余余额，差额永久丢失。

### 1.5 合约验证

```javascript
// 验证实现合约（不是代理！代理不需要验证）
const implAddress = await upgrades.erc1967.getImplementationAddress(stakeAddress);
await hre.run("verify:verify", { address: implAddress, constructorArguments: [] });

// 验证 MetaNodeToken
await hre.run("verify:verify", { address: metaNodeTokenAddress, constructorArguments: [] });
```

只验证**实现合约**和 MetaNodeToken，代理合约本身是 OpenZeppelin 的标准 ERC1967 代理，不需要验证。验证后用户可以直接在 Etherscan 上读写合约。

### 1.6 阶段 3：添加质押池

部署完成后，合约里还没有可用的质押池，需要 Admin 调用 `addPool` 创建。

**添加 ETH 池（`scripts/addPool.js`）**：

```javascript
const tx = await MetaNodeStake.connect(signer).addPool(
    ethers.ZeroAddress,  // stTokenAddress: 0x0 代表 ETH
    500,                 // poolWeight: 权重 500
    100,                 // minDepositAmount: 最小质押额
    20,                  // unstakeLockedBlocks: 锁定期 20 个区块
    true                 // withUpdate: 先更新所有池
);
```

**添加 ERC20 池（`scripts/addERC20Pool.js`）**：

```javascript
// 先部署测试用的 ERC20 代币
const TestERC20 = await ethers.getContractFactory("TestERC20");
const testToken = await TestERC20.deploy("Test Stake Token", "TST", ethers.parseEther("1000000"));

// 再添加为质押池
const tx = await MetaNodeStake.connect(signer).addPool(
    testTokenAddress,          // ERC20 代币地址
    500,                       // poolWeight
    ethers.parseEther("1"),    // minDepositAmount: 1 TST
    20,                        // unstakeLockedBlocks
    true                       // withUpdate
);
```

**为什么第一个池必须是 ETH（`address(0)`）**：

```solidity
// addPool() 中的强制约束
if (pool.length > 0) {
    require(_stTokenAddress != address(0x0), "invalid");  // 后续池不能是 0x0
} else {
    require(_stTokenAddress == address(0x0), "invalid");  // 第一个池必须是 0x0
}
```

Pool[0] 硬编码为 ETH 池。`depositETH()` 只操作 `pool[ETH_PID]`（即 pool[0]），这是一个设计约定，确保原生币质押永远有一个专属入口。

## 二、升级流程

### 2.1 UUPS vs Transparent Proxy

| 对比 | Transparent Proxy | UUPS |
|------|-------------------|------|
| 升级逻辑 | 升级函数在代理合约中 | 升级函数在实现合约中（`_authorizeUpgrade`） |
| Gas 消耗 | 每次调用额外检查 admin | 更省 Gas |
| 复杂度 | 简单 | 更复杂（需正确实现 `_authorizeUpgrade`） |
| 风险 | 代理合约代码量大 | 升级方法写错了会锁死合约 |

### 2.2 upgradeToAndCall 是怎么触发的

这是理解 UUPS 升级最核心的问题。升级函数 `upgradeToAndCall` **在实现合约里**，但最终要修改的是**代理的存储**。这能成功，靠的正是 `delegatecall` 的特性。

#### 完整调用链路

```
npx hardhat run scripts/upgrade.js --network sepolia
      │
      ▼
upgrades.upgradeProxy(proxyAddress, MetaNodeStakeV2)  ← Hardhat 插件
      │
      ├─ ① 编译 V2，对比 storage layout
      ├─ ② 部署新实现合约 → newImplAddress
      │
      ├─ ③ 编码 upgradeToAndCall(newImplAddress, "")
      │      ↓
      │   交易发送到 Proxy 地址
      │      ↓
      │   Proxy 收到 calldata = upgradeToAndCall(newImpl, "")
      │      │
      │      │  Proxy 自己没有这个函数 → 进入 fallback
      │      ▼
      │   fallback: delegatecall → 当前 Impl.upgradeToAndCall(newImpl, "")
      │      │
      │      │  delegatecall 意味着：
      │      │  ├─ 代码在 Impl（老版本）里执行
      │      │  └─ 存储读写全部作用在 Proxy 身上
      │      │
      │      ▼
      └─ ④ Impl.upgradeToAndCall() 执行:
            │
            ├─ _checkProxy()
            │  └→ 验证 address(this) != __self（确认是通过 delegatecall 被调用的）
            │
            ├─ _authorizeUpgrade(newImpl)
            │  └→ MetaNodeStake 中: onlyRole(UPGRADE_ROLE)
            │     有权限 → 继续
            │     无权限 → revert
            │
            └─ _upgradeToAndCallUUPS(newImpl, "")
               │
               ├─ newImpl.proxiableUUID()
               │  └→ 验证新实现实现了 UUPS 接口，返回 IMPLEMENTATION_SLOT
               │     如果没实现 → revert
               │
               └─ ERC1967Utils.upgradeToAndCall(newImpl, "")
                  │
                  ├─ _setImplementation(newImpl)
                  │  └→ SSTORE(IMPLEMENTATION_SLOT, newImplAddress)
                  │     ↑ 就这一行，改写了代理指向的实现地址
                  │       因为 delegatecall 执行，写的是 Proxy 的存储
                  │
                  ├─ emit IERC1967.Upgraded(newImpl)
                  └─ data 为空 → _checkNonPayable()
```

#### 核心源码注解

**入口：`UUPSUpgradeable.upgradeToAndCall()`**（在实现合约中）

```solidity
// @openzeppelin/contracts/proxy/utils/UUPSUpgradeable.sol
function upgradeToAndCall(address newImplementation, bytes memory data) public payable virtual onlyProxy {
    _authorizeUpgrade(newImplementation);              // ① 权限检查
    _upgradeToAndCallUUPS(newImplementation, data);    // ② 执行升级
}

modifier onlyProxy() {
    _checkProxy();
    _;
}

function _checkProxy() internal view virtual {
    if (
        address(this) == __self ||                                    // 不能直接调（必须是 delegatecall）
        ERC1967Utils.getImplementation() != __self                    // delegatecall 的目标必须是当前实现
    ) {
        revert UUPSUnauthorizedCallContext();
    }
}
```

`__self` 是在合约部署时记录的自身地址（`address private immutable __self = address(this);`）。当通过 proxy 的 `delegatecall` 调用时，`address(this)` 是 Proxy 的地址，不等于 `__self`，所以通过。如果有人直接调用 Impl 合约上的 `upgradeToAndCall`，此时 `address(this) == __self`，直接 revert。

**权限检查：`_authorizeUpgrade()`**（在 MetaNodeStake 中重写）

```solidity
// MetaNodeStake.sol
function _authorizeUpgrade(
    address newImplementation
) internal override onlyRole(UPGRADE_ROLE) {}
//                      ^^^^^^^^^^^^^^^^^^^^^^^
//                      只有 UPGRADE_ROLE 能通过
//                      函数体为空：不做额外检查，纯看角色
```

**安全校验 + 实际 SSTORE：`_upgradeToAndCallUUPS()`**

```solidity
function _upgradeToAndCallUUPS(address newImplementation, bytes memory data) private {
    // 验证新实现是合法的 UUPS 合约（实现了 proxiableUUID）
    try IERC1822Proxiable(newImplementation).proxiableUUID() returns (bytes32 slot) {
        if (slot != ERC1967Utils.IMPLEMENTATION_SLOT) {
            revert UUPSUnsupportedProxiableUUID(slot);   // 槽位不对 = 不是标准 UUPS
        }
        // 真正改写的就这一行：
        ERC1967Utils.upgradeToAndCall(newImplementation, data);
    } catch {
        revert ERC1967Utils.ERC1967InvalidImplementation(newImplementation);
    }
}
```

**底层 Storage 写入：`ERC1967Utils.upgradeToAndCall()`**

```solidity
// @openzeppelin/contracts/proxy/ERC1967/ERC1967Utils.sol
function upgradeToAndCall(address newImplementation, bytes memory data) internal {
    _setImplementation(newImplementation);   // → SSTORE(IMPLEMENTATION_SLOT, newImpl)
    emit IERC1967.Upgraded(newImplementation);
    if (data.length > 0) {
        Address.functionDelegateCall(newImplementation, data);  // 同时执行一个初始化调用
    } else {
        _checkNonPayable();
    }
}

function _setImplementation(address newImplementation) private {
    if (newImplementation.code.length == 0) {
        revert ERC1967InvalidImplementation(newImplementation);
    }
    StorageSlot.getAddressSlot(IMPLEMENTATION_SLOT).value = newImplementation;
    //                                                      ^^^^^^^^^^^^^^^^
    //                                                      就是这一行 SSTORE
}
```

### 2.3 为什么必须通过 delegatecall

| 调用方式 | address(this) | 能改 Proxy 存储？ | 能升级？ |
|----------|--------------|-------------------|---------|
| 直接调 Impl | Impl 的地址 | ❌ | ❌ `_checkProxy()` 拦住了 |
| Proxy `delegatecall` → Impl | Proxy 的地址 | ✅ | ✅ |

`delegatecall` 让代码在 Impl 里执行，但存储上下文是 Proxy 的。`upgradeToAndCall` 里那句 `SSTORE(IMPLEMENTATION_SLOT, newImpl)` 因此才能改到 Proxy 的核心槽位，完成升级。

### 2.4 对比：部署 vs 升级，向 Proxy 写入实现地址的路径

| | 部署（第一次绑定） | 升级（替换绑定） |
|---|---|---|
| 时机 | Proxy 构造函数里 | 通过 fallback → delegatecall |
| 调用者 | 插件（部署交易） | Admin（交易到 Proxy） |
| SSTORE 位置 | Proxy 的 constructor | `ERC1967Utils.upgradeToAndCall()` 内部 |
| 权限控制 | 无（谁部署谁说了算） | `_authorizeUpgrade` → `onlyRole(UPGRADE_ROLE)` |
| 安全校验 | 无 | `proxiableUUID()` 验证新实现 |

### 2.5 执行升级的脚本

```bash
npx hardhat run scripts/upgrade.js --network sepolia
```

```javascript
// scripts/upgrade.js（需创建）
const { ethers, upgrades } = require("hardhat");

async function main() {
    const PROXY_ADDRESS = "0x..."; // 当前代理地址

    const MetaNodeStakeV2 = await ethers.getContractFactory("MetaNodeStake");
    const upgraded = await upgrades.upgradeProxy(PROXY_ADDRESS, MetaNodeStakeV2);

    console.log("Upgraded. New implementation:", await upgrades.erc1967.getImplementationAddress(PROXY_ADDRESS));
}

main();
```

**`upgradeProxy()` 内部自动完成**：

```
1. 读取 .openzeppelin/sepolia.json
   └→ 获取当前实现合约地址和 storage layout

2. 编译新的 MetaNodeStake 合约
   └→ 提取新实现的 storage layout

3. 对比新旧 layout（核心安全检查）
   ├─ Slot 顺序一致            ✅
   ├─ 变量类型未变              ✅
   ├─ 变量名可改                ✅（layout 只看 slot 和类型）
   ├─ 新增变量追加到末尾        ✅
   ├─ 删除/重排/改类型          ❌ → 拒绝升级，报 layout incompatible
   └─ 修改继承链的 namespace    ❌ → 同上

4. 部署新实现合约

5. 调用 Proxy 的 upgradeToAndCall()
   └→ delegatecall → 当前 Impl._authorizeUpgrade()（权限检查）
   └→ SSTORE(IMPLEMENTATION_SLOT, newImpl)（替换实现地址）

6. 更新 .openzeppelin/sepolia.json
   └→ 追加新的实现合约记录
```

### 2.6 存储布局安全（Storage Layout）

这是 UUPS 升级最核心的安全约束。从 `.openzeppelin/sepolia.json` 中可以看到已验证的 layout：

```
当前版本 (solc 0.8.22):
Slot 0: startBlock          (uint256)
Slot 1: endBlock            (uint256)
Slot 2: MetaNodePerBlock    (uint256)
Slot 3: withdrawPaused      (bool) + claimPaused (bool) + MetaNode (address)
Slot 4: totalPoolWeight     (uint256)
Slot 5: pool                (Pool[])
Slot 6: user                (mapping)
```

**升级时允许的操作**：

| 操作 | 是否允许 | 原因 |
|------|---------|------|
| 新增状态变量（追加到末尾） | ✅ | 不改变已有变量 slot |
| 修改变量名 | ✅ | slot 位置不变，只是标签 |
| 修改函数逻辑 | ✅ | 不影响存储 |
| 删除变量 | ❌ | slot 编号会错乱 |
| 改变变量类型 | ❌ | 读写长度不一致 |
| 调整变量声明顺序 | ❌ | slot 位置会变 |
| 在继承链中间插入新合约 | ❌ | namespace 排序冲突 |

### 2.7 .openzeppelin 目录的作用

```
.openzeppelin/
├── sepolia.json          ← Sepolia 网络的部署记录
├── unknown-*.json        ← 其他网络的部署记录
└── ...

sepolia.json 内容:
{
  "proxies": [...]   ← 所有代理地址 + 部署 txHash（12 条记录）
  "impls": {         ← 所有实现合约的地址 + storage layout
    "hash1": { address: "0x86B9...", layout: {...} },   ← v1 (solc 0.8.20)
    "hash2": { address: "0x27ad...", layout: {...} }    ← v2 (solc 0.8.22)
  }
}
```

- **不要手动编辑** — 插件自动维护
- **必须提交到 Git** — 团队共享同一份部署状态
- **不可丢失** — 丢了就无法通过插件升级合约
- **不需要手动创建** — 插件首次部署时自动生成

### 2.8 升级的风险点

| 风险 | 后果 | 预防 |
|------|------|------|
| 修改了 storage layout | 存储错乱 → 用户资金混乱 | 升级前插件自动校验 |
| UPGRADE_ROLE 丢了 | 合约永久不可升级 | 多签持有角色 |
| `_authorizeUpgrade` 实现错误 | 所有人或无人能升级 | 函数体保持空实现，靠 role 控制 |
| 新实现有 bug | 资金风险 | 升级前充分测试 |
| 多次升级间 proxy 状态不一致 | .json 文件与实际链上状态脱节 | 不手动编辑 .json 文件 |
